# X Growth Playbook Reference

Source basis:

- `我在X上学到的内容传播方法.txt`
- `X 运营策略.pdf`

This reference is a fixed decision script for Twitter/X content repurposing.
Keep `SKILL.md` small. Use this file as the detailed playbook.

## Fixed Check Sequence

Apply these checks in order before drafting:

1. Choose the content archetype.
2. Define the reader action.
3. Identify which step the post saves for the reader.
4. Identify the available evidence type.
5. Translate the source into external user value.
6. Confirm the save or repost reason.
7. Draft one closed-loop post.

Do not skip ahead to copywriting before this sequence is complete.

## Content Archetypes

Choose exactly one primary archetype:

| Archetype | Use when | Output bias |
|---|---|---|
| Resource entry | The source contains a useful link, collection, directory, template, reading path, dataset, or hard-to-find reference. | Give the value promise, entry point, and save reason. |
| Tool tutorial | The source explains a workflow, setup, prompt, checklist, or practical tool usage path. | Give the scenario, 1-3 steps, proof, and next action. |
| Tool discovery | The source introduces a tool, feature, integration, or AI capability through a concrete task. | Show what task it completes and what signal proves it works. |
| Method steps | The source contains a repeatable process, decision rule, operating cadence, or mistake pattern. | Turn it into a compact method with a clear application scenario. |
| Opinion with proof | The source contains a strong judgment backed by examples, data, screenshots, or constraints. | Lead with the judgment, then attach proof and boundary. |

Avoid choosing an archetype based on topic labels alone. Classify by what the
reader can do after reading.

## Four Saved Steps

Strong X content reduces reader effort. Identify the main saved step:

| Saved step | Meaning | Signal |
|---|---|---|
| Save search | The reader no longer needs to find the source, tool, link, or path. | URLs, directories, collections, source names, search paths. |
| Save understanding | The reader no longer needs to decode a complex idea alone. | Explainers, analogies, simplified steps, beginner framing. |
| Save trial and error | The reader can avoid known pitfalls or start from a tested workflow. | Mistakes, setup notes, prompts, before/after, constraints. |
| Save expression | The reader can repost the idea to represent their own belief or taste. | Strong framing, compact thesis, reusable visual, sharp wording. |

The final post should make this saved step visible.

## Three Evidence Types

Prefer source-backed proof over broad opinion:

| Evidence type | Use when | Examples |
|---|---|---|
| Visible | The source includes or supports a visual proof point. | Screenshot, demo, output, before/after, result image. |
| Clickable | The source gives a path readers can follow. | Tool name, project name, URL, comment entry, search term, repo. |
| Measurable | The source includes concrete numbers or bounded conditions. | Time, cost, user count, install count, steps, conversion, ranking. |

If none is available, do not invent proof. Use a lower-claim opinion format.

## Three Translations

Convert internal or article-centered language into reader value:

1. Announcement -> help.
   - Weak: "We launched X."
   - Strong: "You can use X to finish Y."
2. Capability -> scenario.
   - Weak: "It supports long context."
   - Strong: "It can read a long report and pull competitor changes."
3. Conclusion -> evidence.
   - Weak: "It works well."
   - Strong: "I tried it on this task; here is the output, step, or result."

For product or tool posts, do not publish until at least one translation is
visible in the copy.

## Five-Part X Content Check

A strong X post should behave like a small information product:

1. First-glance value promise: why this matters to the reader.
2. Concrete use case: when the reader would use it.
3. Lowered threshold: step, entry point, explanation, or setup path.
4. Proof: screenshot, number, case, output, constraint, or source detail.
5. Save/repost reason: why the reader would keep it or forward it.

The final post does not need to spell out all five as labels, but the logic
should be present.

## Length And Structure

Default to a standard X post.

For Chinese posts:

- Prefer 120-220 CJK characters when the source supports a practical loop.
- Avoid posts under 40 characters unless the source contains one unusually
  strong standalone judgment.
- Use a closed loop: value -> scenario -> evidence/step -> entry or save reason.

For English posts:

- Prefer compact operator language over literal translation.
- Use one idea, short lines, and a practical reason to act or save.

## Account Context

When publishing account context is available, choose the role deliberately:

| Account type | Best role | Boundary |
|---|---|---|
| Personal account | Experience, explanation, teardown, usage feeling, field notes. | Do not present personal judgment as official product policy. |
| Company account | Accurate update, authority, official link, release context. | Avoid sounding like an ad; show user value. |
| Founder/operator account | Judgment, lesson, experiment, mistake, practical proof. | Do not overclaim results beyond the source. |

## Timing Notes

Use timing only as advisory context, not as a promise:

- Midday can fit tools, tutorials, and resource entries.
- Evening can fit stronger opinions, case breakdowns, and key content.
- Late night can fit high-save developer tools and longer experience posts.

Content quality decides whether the second wave happens. Timing only affects the
first wave.

## Growth And Agent Notes

When the user asks for X operation, growth, or agent-assisted workflows, apply
these rules:

- Use X when real target users, competitors, KOLs, or category conversations are
  active on X.
- Do not judge DM experiments by reply rate alone. Track sent count, reply rate,
  effective conversations, signups, paid conversions, and repeated objections.
- Start controlled DM tests before scaling. Keep single-account volume modest
  until account health and response quality are observed.
- For targeting, prefer users active in the last 3 months around competitors,
  KOLs, or relevant topics over raw follower lists.
- Agent should assist with topic selection, draft variants, data reading, and
  rule extraction. A human should still select, edit, publish, and update the
  next writing rules.

Do not recommend automation that imitates one-to-one relationships at scale or
ignores platform health.
